# Smart Nudge Execution Plan

## Context

The infrastructure for smart nudges is already largely in place. `MemoryManager.generateSmartNudge()` exists, `CheckInWorker` already gates on `smartNudgesEnabled`, `UserPreferencesRepository` already has all the right preference keys, and `SettingsScreen` already has the "Smart Nudges" toggle and the 4-session auto-prompt dialog.

What this plan delivers is the four focused changes needed to make the feature feel intentional, trustworthy, and human — not just functional.

---

## What is already built (do not re-implement)

- `MemoryManager.generateSmartNudge()` — pattern detection across last 4 sessions (recurring theme → day-of-week emotion → dominant emotion fallback)
- `CheckInWorker` — already has the `smartNudgesEnabled` gate and delegates to `generateSmartNudge()` when enabled and 4+ sessions exist
- `UserPreferencesRepository` — `smartNudgesEnabled`, `preferredCheckInTime` (default `20:00`), `hasShownNudgePrompt` all exist
- `NotificationHelper` — `NOTIF_ID_NUDGE` already reserved; `send()` already works
- `SettingsScreen` — "Smart Nudges" toggle, subtitle, and 4-session auto-prompt dialog all built
- `SessionSummaryDao.getLastFourteenSummaries()` — DAO query already exists

---

## Task 1 — Improve nudge message copy

**File:** `app/src/main/java/com/iamashad/meraki/utils/MemoryManager.kt`

**What to change:** The three nudge templates in `generateSmartNudge()` are functional but read as outputs of a pattern-matcher, not as something a person would say. Replace them with warmer, more specific copy that implies the app *heard* the user rather than just *detected a keyword*.

**Current → Target:**

| Pattern | Current | Target |
|---|---|---|
| Recurring theme | `"You mentioned $recurringTheme last time. How are things going with that?"` | `"You've been carrying a lot around $recurringTheme lately — want to check in?"` |
| Day-of-week emotion | `"$dayName evenings can feel heavy sometimes. Meraki is here if you need to talk."` | `"$dayName evenings can feel heavy sometimes. You don't have to carry it alone — Meraki is here."` |
| Dominant emotion fallback | `"You've been feeling $dominantEmotion lately. How are you holding up today?"` | `"You've seemed $dominantEmotion across a few of your recent sessions. How are you holding up?"` |

**Why:** The current copy sounds like it was generated by a system. The revised copy sounds like it was written by someone who cares. The difference is subtle in structure but significant in how it lands on the user.

## Task 2 — Enforce the evening-only window for smart nudges

**File:** `app/src/main/java/com/iamashad/meraki/notifications/CheckInWorker.kt`
**Also touches:** `app/src/main/java/com/iamashad/meraki/repository/UserPreferencesRepository.kt`

**The problem:** Smart nudges currently fire on the same schedule as the daily check-in, which is user-configurable via `preferredCheckInTime`. A user who sets their check-in to 8am would receive a nudge like "You've been carrying a lot around work lately" at 8am — before they've even started work. That's the wrong moment.

**What to change:**

1. Add a new preference key `KEY_SMART_NUDGE_SEND_TIME` to `UserPreferencesRepository` with a hardcoded default of `"20:00"`. This is not user-configurable (no setting exposed for it) — it's a product decision that evening is the right window. The key exists only so it can be overridden later if needed.

2. In `CheckInWorker.schedule()`, when scheduling, use `preferredCheckInTime` for the daily check-in path as before. The smart nudge fires on the same `CheckInWorker` run (not a separate worker), so the right approach is to add a time-of-day guard *inside the smart nudge path*: if the current hour is before 18 (6pm) or after 23 (11pm), skip the nudge and fall through to the regular check-in instead.

**Logic in `CheckInWorker.doWork()`:**

```
// Inside the smart nudge gate block:
val currentHour = Calendar.getInstance().get(Calendar.HOUR_OF_DAY)
val inEveningWindow = currentHour in 18..23
if (!inEveningWindow) {
    // Fall through to default check-in — don't skip entirely
    goto default check-in path
}
```

**Why this approach over a separate worker:** Adding a second worker for nudges introduces scheduling complexity and doubles the WorkManager bookkeeping. The evening window guard is three lines of code and solves the problem without new infrastructure.

## Task 3 — Update the Settings toggle copy and subtitle

**File:** `app/src/main/java/com/iamashad/meraki/screens/settings/SettingsScreen.kt`

**What to change:** The current "Smart Nudges" toggle subtitle reads:

> *"Personalised nudges based on your patterns (available after 4 sessions)"*

Update it to communicate both what the feature does and when it fires:

> *"Personalised evening check-ins based on what you've been sharing lately"*

The parenthetical `(available after 4 sessions)` can be dropped from the subtitle — that context is already communicated by the auto-prompt dialog that appears after the 4th session. Keeping it in the subtitle makes it feel like a limitation rather than a feature.

Also update the auto-prompt dialog text slightly:

**Current:**
> *"You've had 4 or more sessions with Meraki. Smart Nudges can now recognise your patterns and send personalised check-ins — like reminding you about topics you've mentioned before. Would you like to turn them on?"*

**Updated:**
> *"You've been using Meraki for a while now. Smart Nudges can send you a personalised evening check-in when it notices a pattern — like if you've mentioned work or sleep across a few sessions. Would you like to turn them on?"*

**Why:** The original reads like a feature announcement. The updated version explains the *actual behaviour* (evening check-in, specific patterns), which sets expectations correctly and reduces the chance of the feature feeling creepy when it first fires.

## Task 4 — Add a transparency slide to onboarding

**File:** `app/src/main/java/com/iamashad/meraki/screens/register/OnBoarding.kt`

**What to change:** Insert a new onboarding page as the fourth slide (shifting the current "Your Journey Starts Today" to become slide 5, or replacing it — your call on which to keep).

**New page content:**

```
Title: "Meraki Learns With You"

Subtitle: "Over time, Meraki notices patterns in what you share —
like recurring themes or emotions that keep coming up.
When it does, it may send you a gentle evening nudge.
You're always in control — you can turn this off in Settings."

Animation: lottie_onb3 (reuse) or a new calm/observant animation
```

**Why a new page rather than a tooltip or modal:** The 4-session auto-prompt dialog already handles first-time consent *after* the feature becomes available. The onboarding page does something different — it sets the mental model *before* the user has any sessions, so when the nudge eventually arrives, it feels like a natural part of the product rather than something the app started doing unannounced. This is the transparency layer the user specifically asked for.

**Note on the animation asset:** If a new Lottie asset is created for this page, it should feel calm and observant — something that communicates "noticing" rather than "tracking." A gentle eye/sparkle or a wave pattern would work. If no new asset is available, `lottie_onb2` (Understand Your Emotions) is the closest thematic match to reuse temporarily.

---

## Execution order

These four tasks are independent and can be done in any order, but the recommended sequence is:

1. **Task 4 first** (onboarding) — purely additive, zero risk, immediately testable
2. **Task 1** (copy) — contained to one method, no logic changes
3. **Task 3** (settings copy) — two string changes and a subtitle update
4. **Task 2** (evening window guard) — the only task with logic change, do last so it can be tested against the already-updated copy

---

## Out of scope for this iteration

- Signal quality / confidence thresholds on theme detection (agreed: defer)
- User-configurable smart nudge time (the 18:00–23:00 window is fixed for now)
- Per-theme nudge personalisation beyond the three existing pattern types
- Server-side nudge delivery (all nudges remain local WorkManager jobs)
